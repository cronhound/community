# Contributing to Cronhound Community

Thanks for being here. This repo exists because cronhound is better when users and
maintainers talk in the open. Every contribution — a bug report, a feature idea, a
question, a correction to the FAQ — makes the product better for everyone.

## Ways to contribute

### Report a bug
Something broken? [Open a bug report](https://github.com/cronhound/community/issues/new?template=bug_report.yml).
Include what you did, what happened, what you expected. Screenshots help.

If the bug involves your account data (billing, private monitors, specific incident
details), email **support@cronhound.com** instead so sensitive info stays private.

### Request a feature
[Open a feature request](https://github.com/cronhound/community/issues/new?template=feature_request.yml).
Focus on the *problem* you're trying to solve, not just the solution you imagined —
we might know a better path to the same outcome.

Want to brainstorm before committing to a specific idea? Drop it in
[Ideas](https://github.com/cronhound/community/discussions/categories/ideas) first.

### Answer a question
[Q&A discussions](https://github.com/cronhound/community/discussions/categories/q-a)
are the fastest way to help other cronhound users. If you've figured something out,
someone else will need that answer tomorrow.

### Improve the FAQ
Saw a question answered in Q&A that deserves to live longer? Open a PR adding it to
[`faq/`](./faq/). Keep answers short, link to the Discussion for context.

### Add an example
Got a creative cronhound setup? Write it up in [`examples/`](./examples/) and open a PR.
Other users learn faster from real working setups than from abstract docs.

### Show what you built
[Show & Tell](https://github.com/cronhound/community/discussions/categories/show-and-tell)
is for dashboards, integrations, status pages, weird creative uses. We love seeing them.

## Ground rules

- Follow the [Code of Conduct](./CODE_OF_CONDUCT.md). Be kind, be specific, be constructive.
- Search existing issues and discussions before opening new ones.
- Sensitive info (API keys, private URLs, account details) stays in support emails, not public issues.
- Maintainers may edit or move issues that accidentally include internals or sensitive data.

## How PRs get reviewed

For docs, FAQ entries, and examples:
1. Open a PR against `main`
2. A maintainer will review within a few days
3. Merged contributions get credit in the release notes

Code lives in a private repo, so this community repo doesn't accept code PRs — only
content contributions. If you want to contribute code, open a feature request and we'll
figure out a path together.
